# scikit-quri-scaluq Benchmark Report

> Scaluq batched backend の高速化要因と最適バッチサイズの分析結果

## サマリ (TL;DR)

| 観点 | 結論 |
|---|---|
| **どれだけ速い** | `predict_inner` 単体で **17〜24x**、`estimate_grad` は ~2x、`QNNClassifier` 全学習で ~2.4x |
| **何が原因 (主)** | (1) Python ↔ C++ ブリッジ呼び出しを batch 全体で 1 回に集約、(2) `StateVectorBatched` の連続メモリ配置による SIMD ベクトル化 |
| **何が原因 (副次)** | キャッシュ局所性 (state vector が L2 に収まる範囲で線形 scaling) |
| **何が原因では*ない*** | Kokkos の OMP スレッド並列化 (OMP_NUM_THREADS=1〜8 で speedup 一定 ~22x)、warmup によるキャッシュ効果 (cold/warm/subprocess 差は ±10% 内)、`StateVectorBatched` 再利用 vs fresh 確保 (noise 範囲内) |
| **推奨 batch_size** | **256〜512** が sweet spot (15〜22x、メモリ ≤ 数 MB)。**>1024** で L2 cache 越えるとメモリ帯域律速に遷移。**<128** だと Python overhead が残って 2〜8x 止まり |
| **数値正確性** | qulacs vs scaluq の差は `max_diff ≈ 1.28e-15` (浮動小数点誤差レベル) — 完全一致 |
| **state vector メモリ** | `batch × 2^n_qubits × 16 bytes`。Apple Silicon の P-core L2 (16MB) 内で peak、超えると 17% slowdown (batch=4096, n_qubits=8) |

詳細は各セクション参照。Section 7 は 2026-05-25 の再検証で上記結論を裏付け。

## 検証日

- 初回: 2026-04-27
- **再検証**: 2026-05-25 — 主要結論はすべて維持 (詳細はファイル末尾 "7. 再検証 (2026-05-25)" を参照)

## 実行環境

- **CPU**: Apple Silicon (M-series)
  - P-core L1D: 128KB / L1I: 192KB
  - P-core L2: **16MB** (4コア共有)
  - E-core L1D: 64KB / L1I: 128KB
  - E-core L2: 4MB (6コア共有)
  - **L3キャッシュなし**（Unified Memory Architecture）
- **Python**: 3.11
- **scaluq**: via `SimEstimator(use_scaluq=True)`
- **比較対象**: qulacs (`SimEstimator(use_scaluq=False)`)

## 1. 高速化要因の診断結果

| 要因 | 影響度 | 説明 |
|------|--------|------|
| **Pythonオーバーヘッド削減** | 🥇 **最大** | qulacsは512サンプル×3演算子×ゲート数=数千回のPython呼び出し。scaluqは1回のC++呼び出しで全処理完了。 |
| **SIMDベクトル化** | 🥈 **大** | 連続メモリに全サンプルを配置し、CPUのSIMD命令で一気に計算。 |
| **キャッシュ効率** | 🥉 **中** | StateVectorBatched (batch=512, nq=8) は約2MBでL2キャッシュに収まる。qulacsはメモリアクセスが飛び飛び。 |
| **Kokkosスレッド並列化** | ❌ **無関係** | OMP_NUM_THREADS=1でも8でもspeedupは変わらず(21~22x)。 |

### 時間内訳 (batch=512, n_qubits=8)

| 処理ステップ | qulacs | scaluq |
|------------|--------|--------|
| 回路セットアップ | 22.3ms | 4.6ms |
| 推定（実計算） | 851.6ms | 36.5ms |
| **合計** | **873.9ms** | **41.1ms** |

### 数値的正確性

qulacs と scaluq の結果差: **max_diff = 1.28e-15**（浮動小数点誤差レベルで完全一致 ✅）

## 3. 推奨バッチサイズ

| シチュエーション | 推奨batch | 理由 |
|----------------|----------|------|
| **汎用／迷ったら** | **256〜512** | 90%以上の性能を安定して出せる |
| **qubits少ない (≤5)** | **512〜1024** | メモリほぼゼロ、スループット伸び続ける |
| **qubits多い (≥10)** | **256〜512** | L2キャッシュ(~12MB)溢れ防止 |
| **メモリ節約したい** | **128** | それでも8〜10倍のspeedup |

## 4. キャッシュ効果の検証

懸念: 「warmupによるキャッシュ効果でspeedupが水増しされてないか？」

**結果: キャッシュ効果はほぼゼロ ✅**

| 計測方法 | qulacs | scaluq | speedup |
|---------|--------|--------|---------|
| Cold（毎回新規） | 0.8746s | 0.0461s | **18.95x** |
| Warm（従来法） | 0.8697s | 0.0447s | **19.46x** |
| サブプロセス分離 | 0.8637s | 0.0521s | **16.58x** |

Cold/Warm比: qulacs 1.01x, scaluq 1.03x → 誤差範囲

---

## 5. 結論

scaluqバックエンドによる高速化（最大 **20〜23倍**）は**本物**であり、以下の要因による：

1. **バッチ処理によるPythonオーバーヘッドの完全排除**（最大の要因）
2. **連続メモリ配置によるSIMDベクトル化とキャッシュ効率**
3. スレッド並列化（Kokkos OpenMP）はほぼ無関係

最適バッチサイズは **256〜512**。これでspeedup 15〜20倍を安定して達成できる。

## 6. 大バッチ時のキャッシュ溢れ検証 (n_qubits=8)

> 「1024以上のバッチでL2キャッシュ(16MB)が溢れてspeedupが落ちるのでは？」という疑問を検証

| batch | メモリ | speedup | throughput | キャッシュ |
|-------|--------|---------|-----------|-----------|
| 128 | 0.5MB | 11.02x | 6,570 sps | ✅ L2内 |
| 256 | 1.0MB | 17.84x | 10,216 sps | ✅ L2内 |
| 512 | 2.0MB | 20.91x | 12,338 sps | ✅ L2内 |
| 1024 | 4.0MB | 23.04x | 13,636 sps | ✅ L2内 |
| **1536** | **6.0MB** | **23.83x 🏆** | **14,072 sps 🏆** | ✅ L2内 |
| 2048 | 8.0MB | 23.26x | 13,784 sps | ⚠️ L2(16MB)以内 |
| 3072 | 12.0MB | 23.45x | 13,850 sps | ⚠️ L2(16MB)以内 |
| 4096 | 16.0MB | 19.51x 📉 | 11,529 sps 📉 | ❌ L2(16MB)超過 |

### キャッシュ溢れの影響

- P-core L2(16MB)に収まる範囲（〜3072）まではspeedupが落ちず緩やかに上昇し続ける。
- batch=4096でついにL2(16MB)を超え、**17%ダウン**（23.83x → 19.51x）。ただし崖のように急落はしない。
- 理由: 高速化の主因は **SIMDベクトル化＋Pythonオーバーヘッド削減** であり、これらはキャッシュに依存しないため。

### 結論への影響

**256〜512 推奨は変わらない。** これ以上増やしても speedup の伸びは微増（1536でピーク＋14%）で、メモリ使用量が3倍になる割にはメリットが少ない。

## 7. 再検証 (2026-05-25)

`find_optimal_batch.py` / `verify_benchmark.py` / `diagnose_benchmark.py` を再実行して主要結論をチェック。**全部維持**。

### 7-1. バッチサイズ別 speedup (再検証 vs 初回)

| (n_qubits, batch) | 初回 speedup | 再検証 speedup | 初回 sps | 再検証 sps |
|---|---|---|---|---|
| (5, 512) | 17.18x | **23.83x** | 24,039 | 30,471 |
| (5, 1024) | 22.65x | 24.32x | 32,378 | 30,827 |
| (8, 512) | 20.78x | 22.06x | 12,422 | 11,327 |
| (8, 1024) | 22.43x | 23.90x | 13,382 | 12,504 |
| (10, 512) | 19.41x | 19.24x | 5,974 | 4,621 |
| (10, 1024) | 20.27x | 18.11x | 6,216 | 4,428 |

- speedup は 18〜24x で初回と同レンジ。`(5, 512)` で +39% の改善が出ているが、これは qulacs 側の絶対時間が初回より長い (背景プロセスの影響と推測) ことの裏返し。
- n_qubits=10 で throughput が初回比で 20〜30% 下がっている。同じ系統の qulacs 絶対時間も伸びているため、スレッド/Memory 競合 (システム負荷) が支配的だった可能性が高い。speedup 自体は維持。

### 7-2. Cold vs Warm vs Subprocess (cache 起因かの再確認)

batch=512, n_qubits=8 で:

| 計測方法 | 初回 | 再検証 |
|---|---|---|
| Cold | 18.95x | **18.78x** |
| Warm | 19.46x | **21.47x** |
| Subprocess | 16.58x | **17.93x** |

- Cold/Warm/Subprocess の差は依然 ±10% 程度。**キャッシュ依存ではないことが再確認** された。
- `qulacs cold/warm ratio = 0.90x` (=warm の方が遅い) という逆転が見られたが、これも noise 範囲。

### 7-3. Thread scaling (Kokkos OMP が無関係である再確認)

n_qubits=8, batch=512 で `OMP_NUM_THREADS` を 1/2/4/8 にスイープ:

| OMP_NUM_THREADS | qulacs | scaluq | speedup |
|---|---|---|---|
| 1 | 0.9425s | 0.0437s | **21.55x** |
| 2 | 1.0572s | 0.0481s | **22.00x** |
| 4 | 1.0413s | 0.0481s | **21.65x** |
| 8 | 1.0381s | 0.0467s | **22.24x** |

- **speedup は thread 数によらず ~22x 一定**。Kokkos の OMP 並列が高速化に寄与していないことが再確認された。
- 高速化の主因は **(1) Python オーバーヘッド削減 + (2) SIMD ベクトル化** という結論は変更なし。

### 7-4. Time breakdown (batch=512, n_qubits=8)

| 処理ステップ | 初回 qulacs | 再検証 qulacs | 初回 scaluq | 再検証 scaluq |
|---|---|---|---|---|
| 回路セットアップ | 22.3ms | 26.8ms | 4.6ms | 5.9ms |
| 推定 (実計算) | 851.6ms | 1014.6ms | 36.5ms | 42.9ms |
| 合計 | 873.9ms | **1041.4ms** | 41.1ms | **48.7ms** |

- 両側とも絶対時間が ~20% 増。再現性のある測定差 (背景負荷を含む) と考えられるが、両方が同程度上がっているため speedup 比は変わらず。

### 7-5. Numerical correctness

qulacs ↔ scaluq の差は初回・再検証ともに `max_diff ≈ 1.28e-15` (浮動小数点誤差レベル)。**数値一致は完全維持**。

### 7-6. Allocation cost (Fresh vs Reuse `StateVectorBatched`)

| 計測方法 | 時間 |
|---|---|
| Fresh 確保 | 43.8ms ±8.5ms |
| Reuse + set_zero_state | 50.0ms ±7.9ms |
| Reuse speedup | 0.88x |

- Reuse の方がむしろ僅かに遅く出たが、std (±8ms) を考えると noise 範囲。
- いずれにせよ allocation overhead は **<10ms / call**、speedup 23x への寄与は微々たるもの。**主因ではない**。

### 結論 (再検証後)

- 初回レポートの結論はすべて維持: **speedup 17〜24x、Python overhead + SIMD が主因、Kokkos thread parallelism は無関係、cache effect も微小**。
- 数値は ±15% のレンジで再現するため、絶対時間より **speedup 比率と scaling 傾向** を参照するのが信頼性が高い。
